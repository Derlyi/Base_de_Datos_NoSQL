BASES DE DATOS ORIENTADAS A COLUMNAS

¿Qué son las Bases de Datos Orientadas a Columnas?

Las bases de datos orientadas a columnas (BDOCs), también conocidas como column stores en inglés, son un tipo de base de datos NoSQL que optimizan el almacenamiento y la recuperación de datos para análisis de grandes volúmenes de información. A diferencia de las bases de datos relacionales tradicionales, que almacenan datos en filas, las BDOCs almacenan datos en columnas.

CASSANDRA

Cassandra almacena los datos como almacenes clave-valor. Permite definir tablas mediante filas y columnas, pero la estructura tabular no se utiliza en el almacenamiento real. En su lugar, utiliza el modelo de base de datos de columnas anchas, por lo que cada fila de la tabla puede tener un conjunto de columnas diferente. Puede agrupar las columnas en familias de columnas según su tipo de datos o su uso. Cada fila tiene una clave principal que puede utilizar para leer rápidamente los datos de Cassandra.

Funcionamiento:

Cassandra se basa en una arquitectura distribuida y replicada, compuesta por nodos que trabajan juntos para almacenar y procesar datos. Cada nodo almacena una parte del conjunto de datos completo, lo que permite distribuir la carga y aumentar la disponibilidad. En Cassandra, los datos se organizan en keyspaces, que son similares a las bases de datos en las bases de datos relacionales. Dentro de cada keyspace, los datos se almacenan en tablas, que son similares a las tablas en las bases de datos relacionales. Las tablas de Cassandra no tienen un esquema definido, lo que significa que las columnas se pueden agregar y eliminar de forma dinámica. Las filas en Cassandra se identifican por una clave primaria, que puede ser una sola columna o una combinación de columnas. Los valores en Cassandra se almacenan en columnas, que pueden ser de diferentes tipos de datos, como cadenas de caracteres, números, fechas y blobs.

Características:
  1. Escalabilidad horizontal
    Cassandra está diseñada para crecer fácilmente a través de múltiples servidores, ofreciendo una alta tolerancia a fallos. Si un servidor falla, los datos permanecen disponibles en otros nodos de la red.
  2. Alto rendimiento
    Prioriza la lectura y escritura rápida de grandes volúmenes de datos, haciéndola idónea para aplicaciones que manejan información constantemente cambiante.
  3. Disponibilidad siempre activa
    Cassandra está pensada para funcionar sin interrupciones, evitando la pérdida de datos. Cassandra proporciona una API de Java y C++ fáciles de usar para el desarrollo de aplicaciones.
  4. Modelo de datos flexible
    Utiliza un modelo de datos flexible que permite almacenar información en columnas, facilitando la adaptación a estructuras cambiantes.
Desventajas:
  1. Complejidad del modelo de datos
    A diferencia de las bases de datos relacionales con un esquema definido, Cassandra utiliza un modelo de datos flexible sin esquema, lo que puede dificultar el modelado de datos complejos y las relaciones entre ellos.
  2. Consistencia eventual
    El modelo de consistencia eventual de Cassandra puede generar problemas en aplicaciones que requieren una fuerte consistencia de datos en tiempo real. Los cambios en los datos pueden no propagarse inmediatamente a todos los nodos del clúster, lo que puede ocasionar discrepancias temporales en la lectura de datos.
  3. Soporte limitado para transacciones ACID
    Las transacciones ACID en Cassandra tienen limitaciones en cuanto al número de filas involucradas y al aislamiento entre transacciones. Se deben evaluar alternativas como bases de datos relacionales o soluciones de bases de datos híbridas para aplicaciones con demandas transaccionales exigentes.
  4. Dependencia del clúster
    Si un nodo falla, la aplicación puede experimentar interrupciones o degradación del rendimiento hasta que se repare o reemplace el nodo. Se requieren estrategias de alta disponibilidad y tolerancia a fallos para garantizar la continuidad del servicio.
